Since the start of the project, the software architecture for the Raspberry Pi–based handheld device has been designed and the core system infrastructure has been implemented, although the application software itself has not yet been developed. Initial work focused on establishing a reliable graphics pipeline by interfacing a 320×240 SPI display with the Raspberry Pi Zero 2 W. A rendering system was built using Pygame to draw graphical elements to an off-screen surface before converting the 24-bit RGB image into RGB565 format required by the display controller and writing the resulting framebuffer directly to /dev/fb1. This provided a functional low-level graphics backend independent of a desktop environment. In parallel, direct input handling was implemented by reading Linux input events from /dev/input/event0, decoding the kernel input_event structure using Python's struct module, and processing hardware key events without relying on higher-level GUI libraries. This established complete control over both display output and physical input devices at the operating system level. Following the successful implementation of the hardware interface, the project transitioned from a procedural prototype toward a structured software architecture. Early versions passed numerous variables such as menu state, selected options and execution flags between functions. This was replaced with a dedicated State object responsible for maintaining all runtime information, including application execution status, current menu selection, active application identifier, operating mode and application metadata. This significantly reduced parameter passing, centralised runtime information and established a single source of truth for system state. During this stage, several issues related to Python object references, state mutation and function interfaces were identified and resolved, resulting in a cleaner separation between state management and subsystem responsibilities. The project architecture was then reorganised into distinct software modules with clearly defined responsibilities. Rendering operations were isolated within a rendering module responsible solely for drawing user interface elements and updating the framebuffer. Input processing was separated into an input manager responsible for decoding Linux events and translating them into state changes. Application discovery and execution were placed within an application manager responsible for maintaining a registry of installed applications and providing lookup functionality. Shared runtime structures such as State and AppInfo were moved into a dedicated class management module to eliminate duplicated definitions and reduce coupling between components. Throughout this restructuring, multiple circular import issues and package import problems were encountered due to dependencies between modules. These were resolved by redesigning module responsibilities, converting directories into Python packages where appropriate and relocating shared definitions to eliminate recursive imports. A menu system was subsequently implemented to provide the primary user interface for the handheld device. Rather than storing application names independently, the menu now operates on registered application objects identified by unique application IDs. Each application is represented by an AppInfo object containing metadata such as application name, storage size, identifier and execution function. The application manager maintains a registry of available applications and provides lookup functionality to populate menu entries dynamically. This replaced earlier hard-coded menu definitions and established a scalable mechanism for future application expansion. As the architecture evolved, application execution was redesigned to reduce hard-coded behaviour. Earlier implementations relied on conditional statements distributed throughout the codebase to determine which application should execute. This was progressively replaced by an application registration system in which each application provides its own execution function. Execution requests now resolve an application through the registry before dispatching control to the associated application code. This removes the dependency on menu names and prepares the software for future expansion with additional applications requiring minimal modification to core system components. To support multiple operational contexts, a simple state-machine architecture was introduced. The system now differentiates between operating modes such as the main menu and active application execution through a dedicated mode variable stored within the global state object. Input processing was restructured to dispatch events based on the current operating mode rather than treating all keyboard input identically. Menu navigation and application input handling were separated into dedicated handler functions, laying the groundwork for future routing of input events directly to applications. This represents the first implementation of a kernel-style event dispatch model within the software architecture, where a central dispatcher determines the destination subsystem before application-specific processing occurs. Application development infrastructure has also been established. A prototype Notes application was created to validate the application framework, including application registration, execution and rendering. The prototype currently demonstrates independent rendering to the display and application-specific input handling but remains a functional placeholder rather than a complete application. During integration, several architectural issues were identified, including shared selection state between the menu and applications, repeated application dispatch, registration ordering, and inconsistencies between application metadata and execution routing. These issues informed subsequent refinements to the application framework and highlighted areas requiring further separation of application-local state from global system state. Performance optimisation formed an additional aspect of development. Rendering behaviour was examined to reduce unnecessary framebuffer writes while maintaining responsive user interaction. Alternative rendering strategies, including state-change detection and conditional redraws, were evaluated, leading to improvements in framebuffer update efficiency. Although further optimisation remains possible, the rendering pipeline now avoids several earlier inefficiencies while maintaining correct display output on the SPI interface. Overall, the project has progressed from isolated hardware experiments to a functioning embedded software framework capable of directly interfacing with Linux input devices and framebuffer hardware, maintaining centralised runtime state, rendering a graphical user interface, registering and launching applications, and routing input based on operating mode. The current implementation provides the foundational operating environment upon which the intended software applications can now be developed. The remaining work is focused primarily on implementing application functionality, refining application state management, expanding the application framework, improving event routing and rendering efficiency, and completing the user-facing software rather than constructing additional system infrastructure.
